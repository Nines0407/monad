# Monad Blitz 杭州 — Agentic Payment 赛题实现计划

## 项目概述
在 **Monad 链**上构建一个 **Agent 原生安全支付系统**，使 AI Agent 能在用户授权下自主完成链上支付，同时确保安全、可控、可审计。最终产品可集成到 Claude Code / OpenClaw / Codex / Manus 等 AI Agent 环境中。

## 核心要求实现策略

### 1. 去中心化 (必须)
- **非托管方案**：用户自持私钥，Agent 无法接触真实私钥
- **智能合约钱包**：基于 ERC-4337 或类似标准的账户抽象钱包
- **权限委托**：通过 Session Key 机制授予临时权限
- **跨平台支持**：Mac/Linux/Windows，AMD/Intel/Apple M 架构
- **开源**：项目代码完全开源，采用 MIT/Apache 2.0 许可证

### 2. 安全配置 (必须)
- **限额管理**：
  - 单笔交易上限
  - 每日/每周预算限制
  - 基于时间的消费控制
- **白名单机制**：
  - 允许的收款地址
  - 允许的智能合约
  - 允许的 Token 类型
  - 已验证的服务商列表
- **方法级限制**：
  - 特定合约函数的调用权限
  - 禁止危险操作（如 approve 无限额度）
- **人工确认**：
  - 超过阈值时触发人工审批
  - 通过 Telegram/App/浏览器插件通知
- **紧急措施**：
  - 一键暂停所有 Agent 权限
  - 立即撤销 Session Key

### 3. Agent 原生设计 (必须)
- **权限请求流程**：Agent 明确请求支付权限，说明原因和上下文
- **临时授权**：基于任务的短期授权，任务完成后自动失效
- **支付上下文**：支付记录关联任务 ID、Agent ID、用户目标
- **失败处理**：支付失败后的重试逻辑、备选支付路径
- **集成接口**：
  - MCP Server 协议
  - CLI 工具
  - TypeScript SDK
- **判断标准**：系统离开 AI Agent 环境就无法正常使用

### 4. 可审计与可解释 (必须)
- **结构化支付记录**：
  - Task ID, Agent ID, User ID
  - 收款地址、金额、资产类型
  - 支付原因、任务上下文
  - 命中的安全策略
  - 是否经过人工确认
  - 执行结果（成功/失败/错误）
  - 时间戳、交易哈希
- **审计界面**：
  - 按时间、Agent、任务分类查看
  - 导出 CSV/JSON 格式
  - 链上存储关键元数据

### 5. 恢复与权限管理 (必须)
- **钱包恢复**：
  - 社交恢复（多个守护人）
  - 硬件钱包备份
- **权限调整**：
  - 实时修改 Agent 权限
  - 批量权限管理
- **Session Key 轮换**：
  - 定期自动轮换
  - 泄露时手动撤销
- **多 Agent 管理**：
  - 不同 Agent 差异化权限
  - 长期 Agent vs 临时 Agent
  - 角色基础权限控制

## 架构设计

### 系统组件
```
1. 智能合约钱包 (Monad 链上)
   - 主合约：用户账户逻辑
   - Session Key 管理器
   - 策略引擎合约
   - 审计日志合约

2. 本地 Agent 客户端
   - 密钥管理（本地加密存储）
   - Session Key 生成器
   - 策略检查器
   - 交易构造器

3. 策略服务器（可选）
   - 中央策略管理
   - 风险分析引擎
   - 人工审批接口

4. 审计服务
   - 支付记录数据库
   - 查询 API
   - 导出功能

5. 集成接口
   - MCP Server
   - CLI 工具
   - TypeScript/JavaScript SDK
```

### 数据流
```
Agent 请求支付 → 检查本地策略 → 请求 Session Key 签名 → 
→ 构造交易 → 发送到 Monad 网络 → 记录审计日志 → 
→ 返回结果给 Agent
```

### 安全边界
- **私钥永不离开用户设备**
- **Session Key 仅在内存中存在**
- **所有策略在签名前验证**
- **关键操作需要二次确认**

## 加分特性优先级

### 高优先级（推荐先实现）
1. **B. Policy Engine** - 可配置策略层
   - 规则引擎支持复杂条件
   - 实时策略更新
   - 策略模板库

2. **A. Session Key / Delegated Key** - 受限临时密钥
   - 时间限制、金额限制
   - 功能限制、收款方限制
   - 自动失效机制

3. **F. 审计日志 / Payment Receipt** - 结构化支付收据
   - 标准化收据格式
   - 链上/链下存储
   - 可验证性

### 中优先级
4. **E. 人机协同审批** - 分级审批流程
   - 小额自动批准
   - 中额二次确认
   - 高风险人工审批

5. **D. 原生机器支付** - API 按次付费
   - x402 协议集成
   - streaming payment
   - micropayment 支持

### 低优先级
6. **C. 现代登录体验** - zkLogin / Passkeys
   - 与传统安全方案结合
   - 渐进式增强

```typescript

tools: [
  {
    name: "request_payment",
    description: "请求支付权限并执行支付",
    inputSchema: {
      type: "object",
      properties: {
        amount: { type: "string" },
        currency: { type: "string" },
        recipient: { type: "string" },
        reason: { type: "string" },
        taskContext: { type: "string" }
      }
    }
  },
  {
    name: "check_budget",
    description: "检查剩余预算和支付限制"
  }
]
```

### CLI 工具命令
```bash
# 初始化钱包
agent-pay init --network monad-testnet

# 授权 Agent
agent-pay authorize --agent-id "claude-code" --budget "10 USDC" --expiry "24h"

# 查看审计日志
agent-pay audit --task-id "task-123"

# 紧急暂停
agent-pay pause --all
```

### SDK 使用示例
```typescript
import { AgentPay } from '@agent-pay/sdk';

const agentPay = await AgentPay.init({
  walletAddress: '0x...',
  network: 'monad-testnet'
});

const result = await agentPay.requestPayment({
  amount: '1.5',
  currency: 'USDC',
  recipient: '0x...',
  reason: '支付 API 调用费用',
  taskContext: '数据分析任务 ID: 123'
});
```

## 差异化创新方向

当前的 Web3 钱包已从“资产管理工具”转向“链上身份入口”。要在竞争中脱颖而出，关键在于**交互逻辑的范式转移**和**特定场景的深度耦合**，而非功能堆砌。

### 1. 从“资产管理”转向“意图导向 (Intent-Centric)**
- **差异化点**：引入账户抽象 (AA) 和 AI 意图解析
- **创意实现**：用户输入自然语言意图（如“用 USDT 赚取 5% 以上稳定收益”），钱包自动完成跨链、换币、存入协议等所有步骤，前端只显示总余额和任务状态

### 2. 社交契约与“影子账户”系统
- **差异化点**：建立基于社交关系的分级授权体系
- **创意实现**：
  - **亲密账户**：允许特定地址（家人）提取小额应急资金，无需私钥
  - **社交恢复进化**：通过社交关系证明找回钱包，替代助记词
  - **影子地址**：一键生成针对单次交易的临时隐私地址，保护资产主轴

### 3. “离线”与物理世界的交互枢纽
- **差异化点**：强化 NFC 与硬件级安全隔离的移动端体验
- **创意实现**：
  - **“碰一碰”支付/签到**：NFC 技术实现线下加密支付或 DAO 身份验证
  - **遗嘱/定时解锁**：设定逻辑，若钱包一年内未激活，资产自动转移至预设地址

### 4. 游戏化与链上行为勋章 (Proof of Degens)
- **差异化点**：将链上数据可视化为属性
- **创意实现**：钱包内置动态 NFT 形象，外观随链上行为变化（如流动性提供者→“农民”装扮，治理投票者→“议员”权杖），增强身份认同和用户粘性

### 5. 针对开发者/极客的“可编程性”
- **差异化点**：“白帽友好型”或“自动化极客”钱包
- **创意实现**：
  - 允许编写简单脚本（类似 iOS 快捷指令），如“Gas 低于 10gwei 且余额 >1 ETH 时自动质押”
  - 提供强大交易模拟器，清晰展示交易后每一笔代币流向

### 建议的切入路径
不要开发“全能钱包”，而是深耕一个**垂直痛点**：

| 痛点 | 解决方案建议 |
| :--- | :--- |
| **门槛太高** | “无助记词”钱包（邮件登录 + 社交恢复） |
| **信息过载** | “降噪”钱包（只显示重要治理提醒和空投，屏蔽垃圾代币） |
| **安全性担忧** | “策略”钱包（大额转账需要双人签名或 24 小时延迟生效） |

**核心思路**：用户不需要另一个 Metamask，他们需要能**屏蔽复杂性**或**定义身份**的智能管家。

## 风险与挑战

### 技术风险
1. **Monad 链兼容性**：确保合约和交易与 Monad 链特性兼容
2. **Session Key 安全**：防止内存泄露和滥用
3. **策略引擎复杂性**：平衡灵活性与性能

### 安全风险
1. **权限漏洞**：仔细检查所有权限检查点
2. **前端安全**：防止 XSS、CSRF 攻击
3. **私钥管理**：确保本地存储安全

### 开发风险
1. **时间管理**：优先实现核心功能，逐步添加加分特性
2. **测试覆盖**：确保关键路径有充分测试
3. **文档完整性**：提供清晰的使用和集成指南
